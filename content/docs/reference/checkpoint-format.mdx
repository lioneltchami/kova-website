# Checkpoint Format

Kova writes `.progress.json` to the project root after each task completes. This file tracks build state for resumption, powers `kova status` output, and stores cumulative token usage. It is written atomically (write to temp file, then rename) so crashes never produce a corrupt checkpoint.

The file uses the `CheckpointFile` TypeScript type from `src/types.ts`.

## Top-Level Fields

| Field         | Type            | Description                                                                             |
| ------------- | --------------- | --------------------------------------------------------------------------------------- |
| `plan`        | string          | Filename of the plan markdown file (not full path). Example: `"add-user-profiles.md"`.  |
| `started_at`  | ISO 8601 string | Timestamp when `kova build` first started for this plan.                                |
| `status`      | string          | Overall build status. One of: `"in_progress"`, `"completed"`, `"failed"`.               |
| `tasks`       | object          | Map from task ID (string) to `CheckpointTask` object.                                   |
| `token_usage` | object or null  | Cumulative token usage across all completed tasks. Null until the first task completes. |
| `validation`  | object or null  | Quality-engineer validation result. Null until validation completes.                    |

## tasks Map

The `tasks` field maps each task ID (as defined in the plan's `Task ID` field) to a `CheckpointTask` object.

### CheckpointTask Fields

| Field           | Type                    | Description                                                                                |
| --------------- | ----------------------- | ------------------------------------------------------------------------------------------ |
| `status`        | string                  | Task status. One of: `"pending"`, `"in_progress"`, `"completed"`, `"failed"`, `"blocked"`. |
| `agent_type`    | string or null          | Agent type assigned to this task. Null for pending tasks.                                  |
| `model`         | string or null          | Model used (`"haiku"`, `"sonnet"`, or `"opus"`). Null for pending tasks.                   |
| `agent_id`      | string or null          | Unique ID of the agent instance. Used internally for logging. Null until the task starts.  |
| `started_at`    | ISO 8601 string or null | When the agent started executing. Null for pending tasks.                                  |
| `completed_at`  | ISO 8601 string or null | When the agent finished. Null for pending and in-progress tasks.                           |
| `duration_s`    | number or null          | Execution time in seconds. Null until completed.                                           |
| `tokens`        | object or null          | Token counts for this task. Null until the task completes.                                 |
| `tokens.input`  | number                  | Input tokens consumed by this task's agent invocation.                                     |
| `tokens.output` | number                  | Output tokens produced by this task's agent invocation.                                    |
| `tokens.total`  | number                  | Sum of input and output tokens.                                                            |

### Task Status Values

| Status        | Meaning                                                        |
| ------------- | -------------------------------------------------------------- |
| `pending`     | Task has not started. Waiting for dependencies to complete.    |
| `in_progress` | Agent is actively running.                                     |
| `completed`   | Agent finished successfully.                                   |
| `failed`      | Agent failed or timed out. Build stops after the current wave. |
| `blocked`     | Task cannot start because a dependency failed.                 |

## token_usage Object

The `token_usage` field is populated after the first task completes and updated after each subsequent task. It reflects cumulative usage across the entire build, including across resumes.

| Field                 | Type            | Description                                                                          |
| --------------------- | --------------- | ------------------------------------------------------------------------------------ |
| `total_input`         | number          | Sum of all completed task input tokens.                                              |
| `total_output`        | number          | Sum of all completed task output tokens.                                             |
| `total_combined`      | number          | Sum of total_input and total_output.                                                 |
| `cost_estimate_usd`   | number          | Estimated cost in USD based on TOKEN_COSTS rates.                                    |
| `per_task`            | object          | Map from task ID to per-task token summary.                                          |
| `per_task[id].input`  | number          | Input tokens for this task.                                                          |
| `per_task[id].output` | number          | Output tokens for this task.                                                         |
| `per_task[id].model`  | string          | Model used for this task.                                                            |
| `session_start`       | ISO 8601 string | When the build session started (used for window calculations).                       |
| `plan_type`           | string          | Claude plan type from kova.yaml. One of: `"pro"`, `"max5"`, `"max20"`, `"api"`.      |
| `window_allocation`   | number          | Token allocation for the configured plan type. Used to calculate budget percentages. |

## validation Object

Present after the quality-engineer validation task completes. Null before then.

| Field     | Type    | Description                                                                  |
| --------- | ------- | ---------------------------------------------------------------------------- |
| `passed`  | boolean | `true` if all acceptance criteria passed and all quality commands succeeded. |
| `results` | object  | Map from criterion text to boolean (passed/failed).                          |

## File Location and Gitignore

`.progress.json` is always written to the project root. Add it to `.gitignore`:

```
# .gitignore
.progress.json
.progress.json.bak
```

The `.bak` file is written when you start a fresh build over an existing checkpoint. It preserves the previous run's data for reference.

## Reading the Checkpoint Programmatically

The checkpoint is plain JSON and can be read from any language. From TypeScript:

```typescript
import { readFileSync } from "fs";
import type { CheckpointFile } from "kova-cli/types";

const checkpoint = JSON.parse(
  readFileSync(".progress.json", "utf-8"),
) as CheckpointFile;

const completed = Object.values(checkpoint.tasks).filter(
  (t) => t.status === "completed",
).length;

const total = Object.keys(checkpoint.tasks).length;
console.log(`${completed} / ${total} tasks complete`);
```
