# Plan Format

A Kova plan is a structured markdown document that defines the work to be done, which agents will do it, and the order they must execute. `kova plan` generates plans automatically, but you can also write them manually or edit generated plans before running `kova build`.

Plans live in `.claude/tasks/` and use lowercase-with-dashes filenames: `add-user-profiles.md`, `fix-checkout-bug.md`.

## Required Sections

Every valid plan must contain these section headings. The plan validator (configured in `plan_validation` in `kova.yaml`) enforces this at plan creation time.

### Task Description

A paragraph describing the overall task and its scope. Provides context to all agents before they read their individual task definitions.

```markdown
## Task Description

Add user profile editing to the application. Users can update their display name,
bio, and avatar image. Avatar images are stored in Supabase Storage. Profile changes
are reflected immediately across the app without requiring a page refresh.
```

### Objective

Defines what "done" looks like. Should be measurable and unambiguous. The quality-engineer reads this section to evaluate whether the build succeeded.

```markdown
## Objective

Users can navigate to their profile page, edit their display name and bio,
upload a new avatar image (JPEG or PNG, max 5MB), and see changes reflected
immediately. The profile endpoint returns the updated data within 500ms.
```

### Relevant Files

Files the agents will read or modify. Listing files explicitly improves agent accuracy because agents focus on these files rather than exploring the entire codebase.

```markdown
## Relevant Files

- `lib/services/profiles.ts` - Profile service (read/write profile data)
- `app/profile/edit.tsx` - Profile edit screen (create new)
- `components/common/AvatarUpload.tsx` - Avatar upload component (create new)
- `supabase/migrations/` - Add storage bucket policy migration
- `types/database.types.ts` - Add avatar_url to profiles type
```

### Step by Step Tasks

The ordered list of tasks. Each task is a level-3 heading followed by metadata fields and a description. See [Task Metadata Fields](#task-metadata-fields) for field details.

```markdown
## Step by Step Tasks

### 1. Create storage bucket

- **Task ID**: setup-storage
- **Agent**: supabase-specialist
- **Model**: haiku
- **Depends On**: none
- Create a `avatars` storage bucket in Supabase with public read access.
- Add a migration file to create the bucket and set RLS policy allowing authenticated users to upload.

### 2. Build profile API

- **Task ID**: build-api
- **Agent**: backend-engineer
- **Model**: sonnet
- **Depends On**: setup-storage
- Add `updateProfile(userId, data)` and `uploadAvatar(userId, file)` functions to `lib/services/profiles.ts`.
- `uploadAvatar` should upload to the `avatars` bucket and return the public URL.
- Update the profile type to include `avatar_url: string | null`.
```

### Acceptance Criteria

An ordered list of concrete, verifiable criteria. The quality-engineer checks these when validating the build. Write them as assertions, not as tasks.

```markdown
## Acceptance Criteria

1. Users can update display name and bio from the profile edit screen
2. Avatar upload accepts JPEG and PNG files up to 5MB, rejects others
3. Uploaded avatars appear on the profile screen within 2 seconds
4. TypeScript compiles with no errors (`tsc --noEmit` passes)
5. Lint passes (`biome check .` returns exit 0)
6. Profile data persists across page refreshes
```

## Optional Sections

These sections improve plan quality but are not required by default. Required status is configurable via `plan_validation.required_sections` in `kova.yaml`.

### Team Orchestration

Notes for the master orchestrator. Documents which tasks can run in parallel, special coordination requirements, or risks to watch for.

```markdown
## Team Orchestration

Tasks 2 (build-api) and 3 (build-ui) can run in parallel after Task 1 (setup-storage) completes.
Task 4 (build-avatar-component) depends on both Tasks 2 and 3.

Risk: The avatar upload size limit must be enforced both client-side (file input) and
server-side (Supabase storage policy). Both agents must use the same 5MB limit value.
Extract it as a constant in lib/constants.ts and import from both places.
```

### Team Members

A roster of agents assigned in this plan. Used by `kova team-build` to register agents before execution begins.

```markdown
## Team Members

- **supabase-specialist**: Tasks 1 (setup-storage)
- **backend-engineer**: Task 2 (build-api)
- **frontend-specialist**: Tasks 3 (build-ui), 4 (build-avatar-component)
- **quality-engineer**: Task 5 (validate)
```

## Task Metadata Fields

Every task under `## Step by Step Tasks` can include these fields:

| Field         | Required    | Format                          | Description                                                                                      |
| ------------- | ----------- | ------------------------------- | ------------------------------------------------------------------------------------------------ |
| `Task ID`     | Yes         | kebab-case string               | Unique identifier. Used in `Depends On` references. Must be unique within the plan.              |
| `Agent`       | Yes         | Agent type name                 | The specialist agent to assign. See [Agent Types](/docs/reference/agent-types).                  |
| `Model`       | No          | `haiku`, `sonnet`, or `opus`    | Override auto-selected model for this task only.                                                 |
| `Depends On`  | Recommended | Task ID or comma-separated list | Tasks that must complete before this one starts. Use `none` for independent tasks.               |
| `Parallel`    | No          | `true` or `false`               | Mark `true` if this task can run alongside unrelated tasks. Default: determined by dependencies. |
| `Assigned To` | No          | Agent name string               | Named agent instance (used in `team-build` with Agent Teams).                                    |

## Dependency Syntax

### No dependencies (runs first)

```markdown
- **Depends On**: none
```

### Single dependency

```markdown
- **Depends On**: setup-storage
```

### Multiple dependencies (all must complete before this task starts)

```markdown
- **Depends On**: build-api, build-ui
```

### Circular dependencies

Circular dependencies are detected by the plan validator and cause the plan to be rejected. Every dependency chain must have at least one task with `Depends On: none`.

## Complete Example Plan

```markdown
## Task Description

Add user profile editing with avatar upload. Users can update display name,
bio, and avatar. Profile changes are immediately visible across the app.

## Objective

Profile edit screen allows updating name, bio, and avatar image. Avatar
upload accepts JPEG/PNG up to 5MB. Changes persist and appear within 2 seconds.

## Relevant Files

- `lib/services/profiles.ts` - Profile service functions
- `app/profile/edit.tsx` - Profile edit screen (create)
- `components/common/AvatarUpload.tsx` - Avatar upload component (create)
- `supabase/migrations/` - Storage bucket migration
- `types/database.types.ts` - Profile type updates

## Step by Step Tasks

### 1. Create avatar storage bucket

- **Task ID**: setup-storage
- **Agent**: supabase-specialist
- **Model**: haiku
- **Depends On**: none
- Create `avatars` Supabase Storage bucket with public read access.
- Add migration with RLS policy: authenticated users can insert, owner can update/delete.

### 2. Build profile service functions

- **Task ID**: build-api
- **Agent**: backend-engineer
- **Model**: sonnet
- **Depends On**: setup-storage
- Add `updateProfile(userId, data)` to lib/services/profiles.ts.
- Add `uploadAvatar(userId, file)` returning public URL.
- Update ProfileData type to include `avatar_url: string | null`.

### 3. Build profile edit screen

- **Task ID**: build-ui
- **Agent**: frontend-specialist
- **Model**: sonnet
- **Depends On**: setup-storage
- Create app/profile/edit.tsx with name, bio fields.
- Wire to `updateProfile` on form submit.
- Show success/error toast on completion.

### 4. Build avatar upload component

- **Task ID**: build-avatar-component
- **Agent**: frontend-specialist
- **Model**: sonnet
- **Depends On**: build-api, build-ui
- Create components/common/AvatarUpload.tsx.
- Accept JPEG/PNG only, max 5MB, validate client-side.
- Call `uploadAvatar` on selection, update profile `avatar_url`.
- Show current avatar with tap-to-change interaction.

### 5. Validate profile editing

- **Task ID**: validate
- **Agent**: quality-engineer
- **Model**: sonnet
- **Depends On**: build-avatar-component
- Verify all acceptance criteria.
- Run: tsc --noEmit, biome check .
- Confirm avatar upload, profile save, and data persistence.

## Acceptance Criteria

1. Display name and bio can be updated from the profile edit screen
2. Avatar upload accepts JPEG and PNG up to 5MB, rejects other formats and larger files
3. Uploaded avatar appears within 2 seconds
4. TypeScript compiles with no errors
5. Lint passes with no warnings
6. Profile data persists across page refresh
7. All existing profile tests continue to pass

## Team Orchestration

Tasks build-api and build-ui can run in parallel after setup-storage completes.
build-avatar-component requires both. Use the same 5MB size constant in both
client validation and server-side storage policy.

## Team Members

- **supabase-specialist**: setup-storage
- **backend-engineer**: build-api
- **frontend-specialist**: build-ui, build-avatar-component
- **quality-engineer**: validate
```

## Manual Plan Writing

You do not need to use `kova plan` to create plans. Write a plan file directly in `.claude/tasks/` and run:

```bash
kova build .claude/tasks/my-manual-plan.md
```

Manual plans must pass the same validator as generated plans. Run `kova build --dry-run` to validate without executing:

```bash
kova build --dry-run .claude/tasks/my-manual-plan.md
# Validates plan structure and prints task graph
# Does not start any agents
```
