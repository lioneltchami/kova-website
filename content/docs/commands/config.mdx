# kova config

View and edit your `kova.yaml` configuration without opening the file manually. Supports reading the full config, setting individual values by dot-notation key path, and adding project rules or file protection boundaries.

## Usage

```bash
kova config [subcommand] [args]
```

Running `kova config` without a subcommand displays the current configuration.

## Subcommands

| Subcommand                           | Description                                     |
| ------------------------------------ | ----------------------------------------------- |
| `kova config`                        | Display current configuration as YAML           |
| `kova config set <key> <value>`      | Set a configuration value by dot-notation key   |
| `kova config add-rule <text>`        | Add a plain-text rule injected into every agent |
| `kova config add-boundary <pattern>` | Add a glob pattern to `boundaries.never_touch`  |

## Examples

### View Configuration

```bash
kova config
```

Output:

```yaml
version: 1
project:
  name: my-app
  language: typescript
  framework: next
  package_manager: pnpm
models:
  auto: true
  trivial: haiku
  moderate: sonnet
  complex: opus
  planning: opus
quality:
  test: pnpm vitest run
  lint: pnpm biome check .
  typecheck: pnpm tsc --noEmit
  validate_at_end: true
execution:
  default_mode: build
  max_parallel_agents: 4
  task_timeout_seconds: 600
usage_tracking:
  enabled: true
  plan: max5
  warn_at_percent: 80
```

### Set a Value

Use dot-notation to set any field in `kova.yaml`:

```bash
kova config set models.trivial haiku
kova config set models.moderate sonnet
kova config set models.complex opus
```

```bash
kova config set execution.max_parallel_agents 6
kova config set execution.task_timeout_seconds 900
kova config set execution.default_mode team-build
```

```bash
kova config set usage_tracking.plan max20
kova config set usage_tracking.warn_at_percent 75
```

```bash
kova config set project.name "my-redesigned-app"
```

Changes are written immediately to `kova.yaml` and take effect on the next `kova plan` or `kova build`.

### Add a Project Rule

Rules are plain-text instructions injected into every agent's context window. They encode project conventions that agents must follow:

```bash
kova config add-rule "Never modify migration files -- create a new migration instead"
kova config add-rule "All API endpoints must validate input and return typed responses"
kova config add-rule "Use TypeScript strict mode throughout -- never use the any type"
kova config add-rule "All database queries go through lib/services/ -- never call Supabase directly from screens"
```

Rules are appended to the `rules` array in `kova.yaml`:

```yaml
rules:
  - "Never modify migration files -- create a new migration instead"
  - "All API endpoints must validate input and return typed responses"
  - "Use TypeScript strict mode throughout -- never use the any type"
```

### Add a Protected File Boundary

Add a glob pattern to `boundaries.never_touch`. Agents will refuse to modify matching files, enforced by the hook validator:

```bash
kova config add-boundary ".env.production"
kova config add-boundary "secrets/**"
kova config add-boundary "supabase/migrations/**"
kova config add-boundary "infrastructure/terraform/**"
kova config add-boundary "*.lock"
```

Resulting `kova.yaml` entry:

```yaml
boundaries:
  never_touch:
    - ".env*"
    - "secrets/**"
    - "supabase/migrations/**"
    - "infrastructure/terraform/**"
    - "*.lock"
```

## Key Reference

Common dot-notation keys for `kova config set`:

| Key                                | Type    | Example                       |
| ---------------------------------- | ------- | ----------------------------- |
| `models.trivial`                   | string  | `haiku`                       |
| `models.moderate`                  | string  | `sonnet`                      |
| `models.complex`                   | string  | `opus`                        |
| `models.planning`                  | string  | `opus`                        |
| `models.auto`                      | boolean | `true`                        |
| `execution.max_parallel_agents`    | number  | `6`                           |
| `execution.default_mode`           | string  | `build` or `team-build`       |
| `execution.task_timeout_seconds`   | number  | `900`                         |
| `usage_tracking.plan`              | string  | `pro`, `max5`, `max20`, `api` |
| `usage_tracking.warn_at_percent`   | number  | `75`                          |
| `usage_tracking.pause_at_percent`  | number  | `90`                          |
| `quality.validate_at_end`          | boolean | `true`                        |
| `quality.validate_after_each_task` | boolean | `false`                       |

## Configuration File Location

`kova config` always reads and writes `kova.yaml` in the current working directory (your project root). There is no global Kova configuration file -- all settings are per-project.
