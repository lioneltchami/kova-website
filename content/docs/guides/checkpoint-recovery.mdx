# Checkpoint Recovery

Kova writes a checkpoint file after every completed task. If a build is interrupted by a rate limit, network timeout, process crash, or Ctrl+C, you can resume from the exact point of interruption. Completed tasks are not re-run. Token counts from completed tasks carry forward so your budget tracking stays accurate.

## How Checkpoints Work

After each task completes, Kova writes the task result to `.progress.json` in your project root. The write is atomic: Kova writes to a temporary file first, then renames it to the final path. This means a crash during the write never leaves a corrupt checkpoint. On Windows, this uses the NTFS rename operation; on Unix, it uses `rename()`.

The checkpoint file records:

- Which tasks are completed, in-progress, or pending
- Token usage for each completed task
- Timing data for each completed task
- The model and agent type used for each task
- The overall build status

Example checkpoint after 2 of 5 tasks complete:

```json
{
  "plan": "add-user-profiles.md",
  "started_at": "2026-03-14T10:00:00Z",
  "status": "in_progress",
  "tasks": {
    "setup-storage-bucket": {
      "status": "completed",
      "agent_type": "supabase-specialist",
      "model": "haiku",
      "agent_id": "abc123",
      "started_at": "2026-03-14T10:00:05Z",
      "completed_at": "2026-03-14T10:00:52Z",
      "duration_s": 47,
      "tokens": {
        "input": 8200,
        "output": 2100,
        "total": 10300
      }
    },
    "build-upload-api": {
      "status": "in_progress",
      "agent_type": "backend-engineer",
      "model": "sonnet",
      "agent_id": "def456",
      "started_at": "2026-03-14T10:00:53Z",
      "completed_at": null,
      "duration_s": null,
      "tokens": null
    },
    "build-profile-form": { "status": "pending" },
    "build-avatar-component": { "status": "pending" },
    "quality-validation": { "status": "pending" }
  },
  "token_usage": {
    "total_input": 8200,
    "total_output": 2100,
    "total_combined": 10300,
    "cost_estimate_usd": 0.005,
    "session_start": "2026-03-14T10:00:00Z",
    "plan_type": "max5",
    "window_allocation": 88000
  },
  "validation": null
}
```

## The --resume Flag

Resume an interrupted build:

```bash
kova build --resume
```

Kova reads `.progress.json`, finds all tasks with `status: "completed"`, and skips them. Tasks that were `in_progress` when the build was interrupted are re-run from the beginning (partial execution is not reliable). All `pending` tasks run in their normal dependency order.

Output on resume:

```
Resuming build: add-user-profiles.md
Completed tasks (skipping):
  setup-storage-bucket  haiku   47s  10,300 tokens

Resuming from:
  build-upload-api      (was running when interrupted -- restarting)
  build-profile-form    (pending)
  build-avatar-component (pending, blocked by build-upload-api + build-profile-form)
  quality-validation    (pending)

Tokens carried forward: 10,300 / 88,000 (11.7%)
```

## Automatic Resume Prompt

If you run `kova build` and a `.progress.json` already exists from an earlier run, Kova detects it and asks:

```
Previous build found: add-user-profiles.md (interrupted 23 minutes ago)
  2 / 5 tasks completed.

Resume from checkpoint? [Y/n]
```

Answer `Y` to resume or `n` to start fresh. Starting fresh archives the old checkpoint to `.progress.json.bak` so you have a record of what ran.

To skip the prompt and always resume:

```bash
kova build --resume --yes
```

To skip the prompt and always start fresh:

```bash
kova build --no-resume
```

## Token Persistence Across Resumes

Token usage from completed tasks is added to the running total when resuming. The budget percentages shown during the resumed build account for all tokens spent in previous sessions on this plan.

```
Resuming build.
Tokens used in previous session: 10,300
Session window allocation:        88,000
Budget consumed before resume:   11.7%

[running] build-upload-api...
```

If a resumed build crosses a warning threshold (80% or 95%), the threshold applies to the combined total, not just the tokens used in the current session. This prevents unexpectedly exhausting your budget across a resume.

## Checkpoint File Location

`.progress.json` is written to the project root (the same directory as `kova.yaml`). One checkpoint file exists per project at a time. Running a new plan overwrites the previous checkpoint (or prompts to resume if the previous build did not complete).

Add `.progress.json` to `.gitignore` to avoid committing build state:

```
# .gitignore
.progress.json
.progress.json.bak
```

## Clearing Checkpoints

Start a build completely fresh:

```bash
kova build --no-resume    # ignores checkpoint, starts from task 1
```

Or delete the checkpoint manually:

```bash
# Unix / macOS / Linux
rm .progress.json

# Windows
del .progress.json
```

## Common Recovery Scenarios

### Rate limit mid-build

Claude Code rate limits return an error that Kova catches. The checkpoint is written for all tasks that completed before the limit was hit. Wait for the rate limit to clear (typically a few minutes for short limits, up to an hour for sustained limits), then:

```bash
kova build --resume
```

### Network timeout

Same as rate limit recovery. The in-progress task restarts from the beginning.

### Process killed (Ctrl+C or crash)

The last-written checkpoint is intact because writes are atomic. Tasks completed before the kill are preserved. Resume normally:

```bash
kova build --resume
```

### Token budget exhausted

If you paused at the 95% budget threshold, wait for your 5-hour window to refresh, then resume. Kova uses the checkpoint's `session_start` timestamp to estimate when the window resets:

```
Budget exhausted: 88,000 / 88,000 tokens (100%)
Window resets in approximately: 2h 41m

Resume with: kova build --resume
```

## enable_resume Configuration

Enable or disable resume behavior globally in `kova.yaml`:

```yaml
execution:
  enable_resume: true # prompt to resume when checkpoint exists (default)
```

Set to `false` to always start fresh and never prompt. Checkpoints are still written even when `enable_resume` is `false`, so you can still use `--resume` manually if needed.
