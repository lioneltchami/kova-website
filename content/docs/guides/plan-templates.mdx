# Plan Templates

Plan templates pre-configure the task graph, agent assignments, and acceptance criteria structure for the most common development scenarios. Instead of the planning agent building a structure from scratch, templates provide a proven starting point that the agent fills in with your specific requirements.

Use templates when you know the shape of the work before you know the details.

## Using Templates

Pass `--template` to `kova plan` with the template name and your prompt:

```bash
kova plan --template <name> "<your prompt>"
```

Or use `kova run` to plan and build in one step:

```bash
kova run --template feature "add user profiles with avatar upload"
```

Available template names: `feature`, `bugfix`, `refactor`, `migration`, `security`, `performance`.

---

## feature

Use for adding new functionality to an existing codebase. The feature template structures work as three phases: foundation (database and API), core (UI and logic), and polish (tests and validation). Dependencies are explicit so agents cannot start UI work before the API contract is defined.

**When to use:** New screens, new API endpoints, new database tables, user-facing workflows.

```bash
kova plan --template feature "add user profiles with avatar upload"
kova plan --template feature "add team invite flow"
kova plan --template feature "add export to CSV on the reports screen"
```

**What the template provides:**

- Phase 1 (Foundation): Database migration task assigned to `supabase-specialist`, storage or config setup
- Phase 2 (Core): API endpoint tasks assigned to `backend-engineer`, UI component tasks assigned to `frontend-specialist`
- Phase 3 (Polish): Integration test task assigned to `quality-engineer`
- Dependency chain enforced: database task must complete before API; API must complete before frontend; frontend must complete before validation
- Default models: haiku for config tasks, sonnet for API and UI, opus for database schema if RLS policies are involved

**Typical agent assignment:**

| Task type          | Agent               | Default model |
| ------------------ | ------------------- | ------------- |
| Database migration | supabase-specialist | opus          |
| API endpoints      | backend-engineer    | sonnet        |
| UI components      | frontend-specialist | sonnet        |
| Integration tests  | quality-engineer    | sonnet        |

## bugfix

Use for investigating and fixing a specific reported bug. The bugfix template separates root cause analysis from the fix so the debugger can form a theory before any code is changed.

**When to use:** Reported bugs, production incidents, regression fixes, issues with unclear causes.

```bash
kova plan --template bugfix "checkout fails when cart has more than 10 items"
kova plan --template bugfix "password reset email not sent for Gmail addresses"
kova plan --template bugfix "app crashes on iOS 17 when opening the camera picker"
```

**What the template provides:**

- Phase 1 (Investigate): Root cause analysis task assigned to `debugger-detective`. Reads relevant files, traces the execution path, identifies the exact failure point, and documents findings before writing any fix.
- Phase 2 (Fix): Implementation task assigned to the appropriate domain specialist based on where the bug lives (frontend, backend, or database). Narrow scope: targets only the files identified in Phase 1.
- Regression test task: `quality-engineer` verifies the bug is fixed and confirms no related behavior is broken.
- Default models: sonnet for investigation and fix, sonnet for validation.

**Typical agent assignment:**

| Task type               | Agent              | Default model |
| ----------------------- | ------------------ | ------------- |
| Root cause analysis     | debugger-detective | sonnet        |
| Fix implementation      | domain specialist  | sonnet        |
| Regression verification | quality-engineer   | sonnet        |

## refactor

Use for improving code structure without changing external behavior. The refactor template begins with a usage mapping task so the agent knows every callsite before touching anything.

**When to use:** Extracting modules, removing duplication, improving naming, splitting large files, consolidating scattered logic.

```bash
kova plan --template refactor "extract payment logic into a dedicated service"
kova plan --template refactor "consolidate the three user query functions into one"
kova plan --template refactor "split the 800-line Dashboard component into smaller pieces"
```

**What the template provides:**

- Phase 1 (Analyze): Usage mapping task assigned to `code-simplifier`. Identifies all consumers of the code being refactored and documents the current public API.
- Phase 2 (Refactor): Implementation task that introduces the new structure incrementally, migrates consumers, and removes the old code.
- Phase 3 (Validate): Behavior verification task assigned to `quality-engineer`. Confirms that external behavior is unchanged using existing tests or by running the quality commands defined in `kova.yaml`.
- Safe approach: the analysis task output gates the refactoring task. The agent cannot proceed without a documented change plan.

**Typical agent assignment:**

| Task type             | Agent             | Default model |
| --------------------- | ----------------- | ------------- |
| Usage analysis        | code-simplifier   | sonnet        |
| Refactoring           | domain specialist | sonnet        |
| Behavior verification | quality-engineer  | sonnet        |

## migration

Use for database schema changes, dependency upgrades, or API version transitions. The migration template handles the three phases that every migration requires: impact analysis, implementation, and cleanup.

**When to use:** Framework upgrades, database schema changes, removing deprecated APIs, moving from one library to another.

```bash
kova plan --template migration "upgrade from React 18 to React 19"
kova plan --template migration "add created_at column to all tables missing it"
kova plan --template migration "replace axios with native fetch throughout the codebase"
```

**What the template provides:**

- Phase 1 (Impact Analysis): Identifies all files, imports, and call sites affected by the migration. Documents breaking changes and any places requiring manual intervention.
- Phase 2 (Implementation): Applies the migration in dependency order. For database migrations, creates a new migration file and updates the service layer. For dependency upgrades, updates imports and adapts call signatures.
- Phase 3 (Cleanup): Removes deprecated code, updates documentation, verifies build passes.
- Rollback documentation: the plan includes a rollback strategy in the acceptance criteria so the path back is defined before execution starts.

**Typical agent assignment:**

| Task type                | Agent               | Default model |
| ------------------------ | ------------------- | ------------- |
| Impact analysis          | general-purpose     | sonnet        |
| Migration                | domain specialist   | sonnet        |
| Schema changes           | supabase-specialist | opus          |
| Cleanup and verification | quality-engineer    | sonnet        |

## security

Use for security-critical changes. The security template always uses `opus` for both the audit and remediation tasks. It also mandates a review step before any fix is applied, preventing automated changes to authentication or authorization code without human confirmation.

**When to use:** Authentication changes, authorization rules, RLS policy audits, input validation, secrets handling, dependency vulnerability patches.

```bash
kova plan --template security "audit and fix RLS policies on user data tables"
kova plan --template security "add rate limiting to the login endpoint"
kova plan --template security "review and patch XSS vulnerabilities in the comment renderer"
```

**What the template provides:**

- Phase 1 (Audit): Security audit task assigned to `security-auditor` on `opus`. Reviews the target code against OWASP Top 10 and documents all vulnerabilities with severity ratings.
- Review step: audit findings are written to a findings file. The build pauses for human review before proceeding to remediation.
- Phase 2 (Remediate): Fix implementation assigned to `security-auditor` on `opus`. Security-auditor (not a general backend engineer) applies all fixes.
- Penetration verification: a follow-up task attempts to exploit the fixed vulnerabilities to confirm they are closed.

**Important:** Security template tasks always use `opus` regardless of `models.auto` setting. Security correctness is worth the additional cost.

**Typical agent assignment:**

| Task type          | Agent            | Model (forced) |
| ------------------ | ---------------- | -------------- |
| Security audit     | security-auditor | opus           |
| Fix implementation | security-auditor | opus           |
| Verification       | quality-engineer | sonnet         |

## performance

Use for optimization work where you need measurable before-and-after comparison. The performance template begins with profiling rather than optimization, preventing premature fixes to the wrong bottleneck.

**When to use:** Slow API endpoints, high database query times, large bundle sizes, poor Core Web Vitals scores.

```bash
kova plan --template performance "reduce API response time on the feed endpoint"
kova plan --template performance "cut the JavaScript bundle size by 30%"
kova plan --template performance "optimize the slow dashboard query that takes 4 seconds"
```

**What the template provides:**

- Phase 1 (Profile): Profiling task assigned to `performance-optimizer`. Identifies the actual bottleneck, measures baseline metrics, and documents findings before any code changes.
- Phase 2 (Optimize): Implementation task applies targeted optimizations — query indexes, caching, lazy loading, code splitting — based on the profiling findings.
- Benchmark comparison: the final task re-runs the same measurements from Phase 1 and documents the improvement. The acceptance criteria requires a measurable delta, not just "it feels faster."
- Before/after metrics are written to the plan output for future reference.

**Typical agent assignment:**

| Task type              | Agent                 | Default model |
| ---------------------- | --------------------- | ------------- |
| Profiling              | performance-optimizer | opus          |
| Optimization           | domain specialist     | sonnet        |
| Benchmark verification | quality-engineer      | sonnet        |

## Creating Custom Templates

Custom templates can be added to `.claude/tasks/templates/` as markdown files following the [plan format](/docs/reference/plan-format). The filename (without extension) becomes the template name:

```bash
# Creates a template usable as --template stripe-integration
.claude/tasks/templates/stripe-integration.md
```

Custom templates are used the same way as built-in templates:

```bash
kova plan --template stripe-integration "add subscription billing"
```

A custom template is a regular plan file with placeholder comments where the planning agent should fill in specifics. Use comments in the form `<!-- PLANNING AGENT: describe what goes here -->` to guide generation.
