# Interactive Mode

When `kova init` runs in an interactive terminal (a TTY) without any flags, it enters interactive mode. A guided prompt sequence walks you through project configuration and generates a complete `kova.yaml` with your choices.

Interactive mode reduces the time-to-first-plan for new users. Instead of reading the full configuration reference before starting, you answer 7 questions and get a working config.

## When Interactive Mode Activates

Interactive mode activates automatically when two conditions are both true:

1. `stdin` is a TTY (you are at a terminal prompt, not piping input)
2. No flags are passed to `kova init`

If either condition is false, Kova uses defaults without prompting. This makes CI/CD environments safe by default.

```bash
# Interactive mode ON (TTY, no flags)
kova init

# Interactive mode OFF (has flags)
kova init --force
kova init --preset nextjs

# Interactive mode OFF (not a TTY -- piped)
echo "" | kova init
```

## The 7-Step Prompt Sequence

### Step 1: Project Name

```
? Project name (my-app):
```

Auto-detected from the `name` field in `package.json`. If no `package.json` exists, defaults to the current directory name. Press Enter to accept the detected value or type a different name.

The project name is written to `kova.yaml` as `project.name` and is passed as context to the planning agent for every plan in this project.

### Step 2: Detected Stack Confirmation

```
? Detected: TypeScript + Next.js + pnpm + Supabase. Correct? (Y/n)
```

Kova runs auto-detection before the prompt sequence starts. This step shows you what was detected and lets you confirm or skip to manual entry.

If you answer `n`, Kova presents individual selection prompts for language, framework, and package manager. If you answer `Y`, detection results are used directly and Steps 3 is skipped.

Auto-detection reads:

- `tsconfig.json` or `pyproject.toml` for language
- `package.json` dependencies for framework
- Lock files for package manager
- `package.json` dependencies for database, auth, and payments libraries

### Step 3: Database (if not confirmed in Step 2)

```
? Database:
  > Supabase
    Prisma
    Drizzle
    MongoDB (Mongoose)
    None
    Other
```

This selection determines which database specialist agent (`supabase-specialist` vs. `database-administrator`) is assigned as the default for database tasks.

### Step 4: Claude Plan Type

```
? Your Claude plan:
  > Max 5x  (~88,000 tokens per 5-hour window)
    Max 20x (~220,000 tokens per 5-hour window)
    Pro      (~44,000 tokens per 5-hour window)
    API      (unlimited -- pay per token)
```

Sets `usage_tracking.plan` in `kova.yaml`. This determines how Kova calculates your session budget percentage and when budget warnings fire. Choosing the wrong plan type means the percentages shown during builds will be inaccurate.

If you are unsure, select the plan that matches your Claude Code subscription.

### Step 5: Default Model

```
? Default agent model:
  > sonnet  (recommended -- best balance of quality and cost)
    haiku   (fastest and cheapest -- lower quality on complex tasks)
    opus    (highest quality -- most expensive)
    auto    (Kova selects per task based on complexity signals)
```

Sets `models.moderate` in `kova.yaml`. The `auto` option is the default and enables Kova's model tiering system, which assigns haiku/sonnet/opus per task. Choosing a fixed model bypasses tiering and applies that model to all tasks.

Most users should choose `auto`. Choose `sonnet` if you want predictable cost without variability. Choose `haiku` only if you are on a tight budget and understand that complex tasks may produce lower-quality output.

### Step 6: Webhook Notifications

```
? Enable webhook notifications? (y/N)
```

If you answer `y`, a follow-up prompt asks for your Discord or Slack webhook URL:

```
? Discord webhook URL (leave blank to skip):
? Slack webhook URL (leave blank to skip):
```

URLs are written to `kova.yaml` under `notifications.on_build_complete`. You can add or change URLs later by editing `kova.yaml` directly or with `kova config set notifications.on_build_complete.discord <url>`.

### Step 7: Protected Files

```
? Files agents should never modify (comma-separated globs):
  .env*, *.lock, node_modules/**
```

Pre-filled with the defaults from `boundaries.never_touch`. Add any additional patterns separated by commas. These globs are written to `kova.yaml` and enforced by the hook validator on every agent file write.

Common additions:

```
secrets/**, *.pem, infrastructure/**, terraform/**
```

## After the Prompts

Kova creates the `.claude/` directory and generates `kova.yaml`, then prints a summary:

```
Created:
  .claude/commands/team-plan.md
  .claude/commands/build.md
  .claude/commands/team-build.md
  .claude/skills/session-management/SKILL.md
  .claude/skills/sub-agent-invocation/SKILL.md
  .claude/hooks/Validators/validate-new-file.mjs
  .claude/agents/agent-rules.json
  kova.yaml

Next steps:
  1. Review kova.yaml
  2. Run: kova plan "your task description"
  3. Review the plan, then: kova build
```

## Skipping Interactive Mode

Pass any flag to use defaults without prompting:

```bash
kova init --force           # overwrite existing .claude/ with defaults
kova init --merge           # merge with existing .claude/, preserve edits
kova init --preset nextjs   # use the Next.js preset without prompting
kova init --dry-run         # show what would be created, create nothing
kova init --no-detect       # skip auto-detection, use all defaults
```

## Re-running Interactive Mode

You can re-run `kova init` on an existing project. If `.claude/` already exists, Kova asks:

```
.claude/ already exists. What would you like to do?
  > Merge (keep my customizations, update templates only)
    Overwrite (replace everything with fresh defaults)
    Cancel
```

Selecting Merge updates the command and hook templates to the current version while preserving your `kova.yaml`, `agent-rules.json`, and any custom skill files.
