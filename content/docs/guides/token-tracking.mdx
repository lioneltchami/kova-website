# Token Tracking

Kova tracks token consumption at the task level and across the entire build session. Every completed task shows how many tokens it used. At the end of a build, you get a full breakdown by task, model, input vs. output, and estimated cost. Budget warnings prevent runaway spending before it happens.

## How Tracking Works

Claude Code does not expose a real-time token API. Kova tracks usage by parsing the JSONL session files that Claude Code writes locally after each agent invocation. Each record includes input tokens, output tokens, cache reads, and cache creation tokens:

```json
{
  "modelUsage": {
    "inputTokens": 12500,
    "outputTokens": 3200,
    "cacheReadInputTokens": 8000,
    "cacheCreationInputTokens": 1500
  },
  "totalCostUSD": 0.042
}
```

Kova sums these values per task and persists the running total in the checkpoint file so token counts survive build interruptions and are accurate on `--resume`.

## Per-Task Display Format

After each task completes, Kova prints a token summary line:

```
[done]  build-upload-api    backend-engineer    sonnet    2m 18s
        Tokens: 18,420 input + 4,210 output = 22,630 total
        Session: 45,280 / 88,000 used (51.5%)  |  42,720 remaining
        Est. cost: $0.08 this task  |  $0.19 total build
```

The session percentage is calculated against your plan's 5-hour window allocation (see Plan Types below). The remaining count lets you estimate whether the build will fit within the current window.

Disable per-task output without disabling tracking:

```yaml
usage_tracking:
  show_per_task: false
  show_build_summary: true # summary still appears at the end
```

## Budget Warning Thresholds

Two thresholds trigger warnings during a build. Both are configurable.

**80% warning** -- Kova prints a warning but continues automatically:

```
  WARNING: Session budget at 82% (72,160 / 88,000 tokens used)
  Remaining tasks: 2 (est. ~25,000 tokens)
  This build may exceed your 5-hour window limit.
```

**95% pause** -- Kova stops and waits for confirmation before continuing:

```
  PAUSED: Session budget at 96% (84,480 / 88,000 tokens used)
  Remaining tasks: 1 (est. ~12,000 tokens)

  Options:
    [C] Continue anyway
    [P] Pause and resume later  (checkpoint saved)
    [S] Switch remaining tasks to haiku to save tokens
```

If you choose P, the checkpoint is saved immediately. Run `kova build --resume` when your 5-hour window has refreshed.

If you choose S, Kova rewrites the remaining task model assignments to `haiku` in the checkpoint and continues. This saves roughly 80-90% of the token cost for those tasks at a quality tradeoff.

## Plan Types and Window Allocations

Claude Code uses a rolling 5-hour window, not a fixed reset. Tokens consumed 5 hours ago become available again continuously. There is no single "reset time" to wait for.

| Plan                | 5-hour window allocation | Weekly cap             |
| ------------------- | ------------------------ | ---------------------- |
| Pro (~$20/mo)       | 44,000 tokens            | Moderate               |
| Max 5x (~$80/mo)    | 88,000 tokens            | High                   |
| Max 20x (~$320/mo)  | 220,000 tokens           | 40 Opus hours          |
| API (pay-per-token) | Unlimited                | None (billed directly) |

These allocations are approximate. Claude Code does not publish exact numbers, and they can change. Kova uses the values from `PLAN_ALLOCATIONS` in its constants, which are updated with each release.

Configure your plan type so Kova calculates percentages correctly:

```yaml
usage_tracking:
  plan: "max5" # pro | max5 | max20 | api
```

If you use `api`, Kova tracks tokens and costs but does not enforce a budget limit. The `warn_at_percent` and `pause_at_percent` settings are ignored for API plan users.

## Token Cost Rates

Kova uses these per-model rates to estimate cost (approximate 2026 pricing):

| Model  | Input per 1M tokens | Output per 1M tokens |
| ------ | ------------------- | -------------------- |
| haiku  | $0.25               | $1.25                |
| sonnet | $3.00               | $15.00               |
| opus   | $15.00              | $75.00               |

Cost estimates in Kova output are labeled "Est." because exact pricing depends on your Anthropic plan, regional pricing, cache hit rates, and any promotional pricing applied to your account. Treat estimates as directional, not exact.

## kova.yaml Configuration

The full `usage_tracking` section with all defaults:

```yaml
usage_tracking:
  enabled: true # set false to disable all token tracking
  plan: "max5" # pro | max5 | max20 | api
  show_per_task: true # show token line after each completed task
  show_build_summary: true # show full table at build end
  show_cost_estimate: true # include cost lines in output
  warn_at_percent: 80 # print warning at this session percentage
  pause_at_percent: 95 # pause and prompt at this session percentage
```

To track tokens without showing them during the build (useful in CI):

```yaml
usage_tracking:
  enabled: true
  show_per_task: false
  show_build_summary: false # summary still written to checkpoint file
```

Token data is always written to the checkpoint file regardless of display settings. `kova status` can show it at any time.

## Viewing Token Usage After a Build

```bash
kova status
# Token usage: 105,330 used  (119.7% of 88,000 window -- exceeded)
# Est. cost: $0.34
# Next window available in: ~2h 43m

kova status --json
# { "token_usage": { "total_combined": 105330, "cost_estimate_usd": 0.34, ... } }
```

## Token Persistence Across Resumes

When a build is interrupted and resumed with `kova build --resume`, token counts from completed tasks carry forward:

```
Resuming build: user-profile-editing
Tokens from previous session: 45,280
Current session: 0
Combined total: 45,280 / 88,000 (51.5%)
```

This means your budget warning thresholds account for all tokens spent on this plan, including those from previous sessions. A build that spans two sessions still respects the single-session window limit.
